# 每日简报 + Idea 发现

从指定目录中读取某个日期的所有日志文件，交付整合简报，同时挖掘可工程化的想法并持久化到 ideas 目录。

## 触发条件

用户请求某个目录中特定日期的每日简报 / 每日摘要 / daily brief / discover ideas。

## 执行步骤

1. **确定源目录** -- 如果用户未指定，使用已知的默认日志目录（如 `~/.hermes/human-like/thoughts/`）。

2. **列出匹配日期的文件** -- 搜索匹配以下模式的文件：
   - `YYYY-MM-DD.md`
   - `YYYY-MM-DD-*.md`
   优先用 `terminal` 枚举：`ls {source_dir}/YYYY-MM-DD*`。`search_files` 对日期模式（含连字符）可能返回空结果，不要依赖它。

3. **没有匹配文件** -- 明确告知用户：在 {目录} 中未找到 YYYY-MM-DD 的日志文件。不生成任何产出。

4. **读取所有匹配文件** -- 对每个文件使用 `read_file(path=..., limit=9999)`。注意优先级：
   - `YYYY-MM-DD.md` 是主日志，优先级最高
   - `YYYY-MM-DD-{SLUG}.md` 是子条目（例如 autonomous-exploration、micro-trigger、dream-thinking）

5. **交付简报** -- 将内容整合成一份连贯的简报：
   - 如果文件有明确主题，按主题/来源分组
   - 如果文件按时间顺序或交错，按逻辑顺序呈现
   - 高亮关键决策、发现或待办事项
   - 保持简洁 -- 用户要的是信号，不是回放

6. **挖掘可工程化的 Idea** -- 读完所有日志后，进行深度分析：
   - **想法提取**：识别文本中的新概念、未探索的方向、"如果……会怎样"类型的思考
   - **工程化评估**：评估每个想法的技术可行性、依赖关系、复杂度
   - **模式发现**：跨条目寻找重复出现的主题
   - **关联推理**：将不同日志中的想法连接，看是否能组合成更大的项目
   - 结合 human-like 项目上下文（agent 系统、autonomous AI、dream-thinking 等）
   - 如果日志内容没有产生有价值的想法，诚实标注为空，不要强行凑数

7. **写入 Idea 文档** -- 保存到 `~/.hermes/human-like/ideas/YYYY-MM-DD-ideas.md`，格式如下：

```markdown
# Ideas: YYYY-MM-DD

来源：{列出所有参与分析的文件名}
生成时间：{当前时间戳}

---

## 💡 Idea 1: {标题}

**来源条目**: {来自哪个日志文件的哪段内容}
**工程化可行性**: 高/中/低
**复杂度**: 小/中/大
**涉及领域**: {如 agent architecture, UX, data pipeline, etc.}

### 描述
{对这个 idea 的清晰描述，2-5 句话}

### 工程化路径
- 关键步骤 1
- 关键步骤 2
- ...

### 潜在挑战
- {技术或资源上的挑战}

---

## 💡 Idea 2: {标题}
...
```

如果有跨日志的模式发现，文档末尾增加 `## 🔁 反复出现的主题` 段落。
如果有可组合的想法，增加 `## 🔗 组合想法` 段落。

8. **汇报** -- 在最终回复中告知用户发现了多少个 idea，简要列出标题和可行性，并指明文件保存路径。

## 输出文件命名

Idea 文档保存到：`~/.hermes/human-like/ideas/YYYY-MM-DD-ideas.md`

如果同一天已存在该文件，**询问用户**是否覆盖，不要默默覆盖。

## 规则

- **绝对不能修改源文件。** 日志目录是只读的。
- **Idea 文档只写入 `~/.hermes/human-like/ideas/` 目录。**
- 不生成摘要文件或简报文件 -- 简报直接在最终回复中交付，持久化的只有 idea 文档。
- 每个 idea 必须有明确的工程化评估（可行性 + 复杂度）。
- 不要生成泛泛而谈的"想法"（如"可以做更好的 UX"）-- 要具体、可执行。
- 如果文件很大，聚焦关键部分，不要 dump 原始内容。
- 如果多个文件覆盖重叠主题，在汇总中去重。
- 始终注明有哪些源文件参与了分析（列出文件名，方便追溯）。

## 常见陷阱

- 某些日期可能没有日志文件 -- 优雅处理，不要报错崩溃。
- 日期格式是 ISO 8601（`2026-04-05`），不是本地化格式。
- 带后缀的文件应标注来源（例如 "来自 autonomous-exploration: ...")。
- 如果同一天有多个 cron 任务生成的文件，用户通常想要合并视图，而不是逐文件罗列。
- 某些日期可能只有一两个小文件，内容稀少 -- 不要强行生成垃圾 idea。
- 日志可能包含大量自我反思而非具体想法 -- 重点挖掘后者。
- 同一天可能有多个 autonomous-exploration 文件（带 -2, -3 后缀），代表不同时段的思考，需分别阅读。
- 文件名中的日期可能和内容中的时间不一致（跨日写入），以文件名为准。
